# Security Audit Report

**Date:** March 19, 2026  
**Scope:** Full codebase — backend (Go/Gin/Huma) and frontend (Vue 3/TypeScript)  
**Reference:** OWASP Top 10

---

## Findings

| ID  | Severity     | Issue                                                                                                                                                                                                                                                                                                                                                                                      | File Path                                                                                                                                                                            | Line Number(s) | Recommendation                                                                                                                                                                                                                                           |
| --- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Critical** | **Open Redirect + JWT Exfiltration** — `/auth/login/:provider` accepts any `http`/`https` URL as `redirect_to`. After OAuth completes the backend redirects to `redirectTo + "#token=" + jwt`, sending the user's JWT to an arbitrary attacker-controlled domain.                                                                                                                          | [backend/web/routes/auth.go](backend/web/routes/auth.go#L72-L88)                                                                                                                     | 72–88, 137–141 | Before storing `redirect_to` in session, assert it starts with `os.Getenv("APP_BASE_URL")`. Reject the login request with 400 if it does not.                                                                                                            |
| 2   | **High**     | **JWT stored in `localStorage`** — All session JWTs are written to `localStorage`, which is readable by any JavaScript executing in the page origin. A single XSS gadget compromises every active session.                                                                                                                                                                                 | [frontend/src/stores/auth.ts](frontend/src/stores/auth.ts#L28-L30)                                                                                                                   | 28–30, 38      | Prefer `HttpOnly` `SameSite=Strict` cookies set by the server. If staying with fragment-to-JS flow, use `sessionStorage` as a minimum; storage in `localStorage` survives tab closure and makes token theft much easier after XSS.                       |
| 3   | **High**     | **No rate limiting on any endpoint** — Login, registration (`POST /auth/register`), URL creation (`POST /user/urls`), and redirect (`GET /r/{shortcode}`) are all unthrottled. Enables brute-force enumeration of registration tokens, DoS of the DB, and redirect scraping. AGENTS.md §8 acknowledges this is required before public release.                                             | [backend/web/routes/auth.go](backend/web/routes/auth.go), [backend/web/routes/urls.go](backend/web/routes/urls.go), [backend/web/routes/redirect.go](backend/web/routes/redirect.go) | —              | Add Gin middleware for per-IP rate limiting (e.g., `golang.org/x/time/rate` or `github.com/ulule/limiter`). Apply stricter limits to unauthenticated endpoints.                                                                                          |
| 4   | **High**     | **Session JWT and pre-registration JWT share the same signing secret** — Both `CreateJWT` and `CreatePreRegToken` use `JWT_SECRET`. `ParseJWT` accepts any token signed with that secret; a pre-reg token submitted as a Bearer header yields `JWTClaims{UserID: 0}`, which passes middleware and enters all protected handlers with a zero user ID.                                       | [backend/web/routes/jwt.go](backend/web/routes/jwt.go#L59-L92)                                                                                                                       | 59–92, 96–120  | Use distinct secrets per token type, or embed a mandatory `typ` claim in both token types and validate it explicitly in each parse function before accepting the claims.                                                                                 |
| 5   | **High**     | **JWT exposed in URL fragment (`#token=…`)** — Even on legitimate same-origin redirects, the raw JWT appears in browser history, is accessible via `window.location.hash` by any first- or third-party script, and may leak through browser extensions.                                                                                                                                    | [backend/web/routes/auth.go](backend/web/routes/auth.go#L137-L141)                                                                                                                   | 137–141        | Exchange the token via a short-lived one-time code instead: backend redirects to `redirectTo?code=<opaque_code>`, the frontend immediately POSTs the code to a token-exchange endpoint to receive the JWT over a JSON body, and the code is invalidated. |
| 6   | **Medium**   | **In-memory JWT blacklist is lost on server restart** — Logged-out tokens remain valid for up to 24 h after any restart, allowing session resumption post-logout.                                                                                                                                                                                                                          | [backend/web/routes/blacklist.go](backend/web/routes/blacklist.go)                                                                                                                   | 1–80           | Persist the blacklist in the database (or Redis if added) so it survives restarts. A simple `jwt_blacklist(jti TEXT, expires_at TIMESTAMPTZ)` table with a periodic purge is sufficient.                                                                 |
| 7   | **Medium**   | **No HTTP security response headers** — Neither Gin nor the Caddyfile set `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, or `Permissions-Policy`. Absence of CSP significantly raises XSS impact; absence of `X-Frame-Options` allows clickjacking.                                                                                            | [deploy/Caddyfile](deploy/Caddyfile)                                                                                                                                                 | —              | Add a `header` block in the Caddyfile (or a Gin middleware) that sends at minimum: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and a strict CSP.                                     |
| 8   | **Medium**   | **DNS-based SSRF bypass in URL validator** — `ValidateLongUrl` only rejects hostnames that parse as a private literal IP. A hostname such as `internal.attacker.com` resolving to `192.168.x.x` passes validation; at redirect time the user's browser is sent to the private address. This enables targeted attacks against company networks of authenticated users.                      | [backend/business-logic/url_validator.go](backend/business-logic/url_validator.go#L36-L47)                                                                                           | 36–47          | Resolve the hostname with `net.LookupHost` (non-cached, with a timeout) and apply the same private-IP check to all resolved addresses before accepting the URL.                                                                                          |
| 9   | **Medium**   | **`gin.Default()` in production logs all request/response metadata** — `gin.Logger()` (included by `gin.Default()`) writes every path, query string, and status code to stdout. OAuth callback query params (error codes, provider names) are captured; if logging infrastructure is compromised these expose auth-flow metadata. `gin.Recovery()` may emit partial request data on panic. | [backend/main.go](backend/main.go#L62)                                                                                                                                               | 62             | Replace `gin.Default()` with `gin.New()` and attach only `gin.Recovery()`. Implement a `slog`-based request logger that explicitly selects which fields to log and masks any sensitive query params.                                                     |
| 10  | **Low**      | **Shortcode base62 generation has modulo bias** — `int(b) % 62` with a uniformly random byte (0–255) is not uniform: `256 mod 62 = 8`, so characters at positions 0–7 in the charset appear with probability 5/256 ≈ 1.95 % vs 4/256 ≈ 1.56 % for the remaining 54 characters. Over large volumes this slightly reduces the effective entropy of generated shortcodes.                     | [backend/business-logic/shortcode.go](backend/business-logic/shortcode.go#L47-L52)                                                                                                   | 47–52          | Use rejection sampling: discard bytes ≥ 248 (`4 × 62`) and redraw, or switch to `math/big`-based uniform sampling.                                                                                                                                       |

## Priority Summary

| Priority                      | IDs                | Action                                                                                                                                                                     |
| ----------------------------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Fix immediately               | #1                 | Single-line server-side guard: validate `redirect_to` starts with `APP_BASE_URL` before storing in session. Closes both the open redirect and the JWT exfiltration vector. |
| Fix before any wider exposure | #4                 | Add a `typ` claim to both JWT types and validate it in each parser. Low effort, high impact.                                                                               |
| Fix before production         | #2, #3, #5, #6, #7 | Token storage strategy, rate limiting, secure-header policy, blacklist persistence.                                                                                        |
| Improve when time allows      | #8, #9, #10        | DNS-resolve SSRF check, structured request logging, rejection-sampling in shortcode generator.                                                                             |
