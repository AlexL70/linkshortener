| 7   | Low      | **Modular bias in shortcode generation weakens unpredictability.** `GenerateShortcode` maps each random byte to a Base62 character via `int(b) % 62`. ## Additional Issues (beyond top 10)

The following lower-severity observations were identified but did not make the top 10:

- **`ValidateCustomShortcode` uses `HasPrefix` for reserved words** ([backend/business-logic/shortcode.go](backend/business-logic/shortcode.go), L57–L61): shortcodes beginning with reserved words (e.g., `"authXYZ"`) are incorrectly rejected. This is a UX bug rather than a security vulnerability but could be considered a logic flaw.
- **`MICROSOFT_CLIENT_ID`/`FACEBOOK_CLIENT_ID` required but providers not registered**: These variables are in `requiredVars` forcing operators to supply placeholder secrets for unimplemented providers, increasing the attack surface if placeholder values are weak.
- **Dev-mode open-redirect bypass**: In dev mode the `redirect_to` same-origin check is completely skipped ([backend/web/routes/auth.go](backend/web/routes/auth.go), L88–L101), allowing any valid HTTP/HTTPS URL as a post-auth redirect. If dev mode is accidentally deployed to a public network this becomes a full open redirect + credential (pre-reg token) leak.
- **Auth callback does not re-validate `redirect_to` origin**: The redirect URL is read back from the session at callback time without re-asserting the same-origin constraint that was applied at login time. The session is signed so tampering is not possible, but defense-in-depth would re-validate.
